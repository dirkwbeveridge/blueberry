## Guardrails

**PRIVACY_BOUNDARY (non-negotiable product promise):** Push payloads must never contain raw health-log fields: `symptoms`, `mood`, `energy_level`, `notes` (raw), or `weight_kg`. Permitted payload body: a shared sentence Mom explicitly chose to share (`shared_note`), an appointment title, a todo title, or a generic signal string ("She logged a check-in"). This is enforced server-side by `assertNoHealthData` in `supabase/functions/shared/pushPayload.ts` and must be called before every `fetch` to the Expo Push API.

**No service-role key on the client.** The Supabase service-role key exists only inside Edge Functions as the `SUPABASE_SERVICE_ROLE_KEY` environment variable. Never reference it in any file under `app/`, `hooks/`, `lib/`, `store/`, or `components/`.

**Quiet hours are server-side.** The Edge Functions check quiet hours against the recipient's local time before dispatching. Client-side quiet-hours logic is additive UX polish, not the enforcement layer.

**expo-notifications only.** Do not import `@apple/apns` or any FCM SDK. All token handling and notification scheduling goes through `expo-notifications`.

## Schema Migration

Run in Supabase SQL Editor after reviewing:

```sql
-- Phase 7c: Push Notifications
-- Adds push token storage and notification preferences to the users table.
-- Adds share_with_partner flag to health_logs for opt-in partner check-in events.

ALTER TABLE users
  ADD COLUMN IF NOT EXISTS push_token               text,
  ADD COLUMN IF NOT EXISTS notif_appointments       boolean NOT NULL DEFAULT true,
  ADD COLUMN IF NOT EXISTS notif_partner_checkin    boolean NOT NULL DEFAULT true,
  ADD COLUMN IF NOT EXISTS notif_todo_assigned      boolean NOT NULL DEFAULT true,
  ADD COLUMN IF NOT EXISTS notif_kick_reminder      boolean NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS quiet_start              int     NOT NULL DEFAULT 21,
  ADD COLUMN IF NOT EXISTS quiet_end                int     NOT NULL DEFAULT 7;
-- quiet_start / quiet_end are local-time hours (0-23). Default: 9 pm to 7 am.

ALTER TABLE health_logs
  ADD COLUMN IF NOT EXISTS share_with_partner  boolean NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS shared_note         text;
-- shared_note is the ONLY health-log field permitted in a push payload.
-- It is a sentence Mom explicitly typed to share. Raw symptom/mood/energy/notes
-- fields remain off-limits even when share_with_partner is true.

-- Index: Edge Function queries partner token by household lookup
CREATE INDEX IF NOT EXISTS idx_users_push_token ON users(household_id) WHERE push_token IS NOT NULL;
```

Also update `supabase-schema.sql` (the canonical schema file) to include these columns so the file stays as single source of truth.

## Dependency Note

Full end-to-end dispatch testing (token → Edge Function → APNs/FCM → device notification) requires at least one real device with the app installed and a valid Expo project ID configured in `app.json` under `extra.eas.projectId`. Plan for a device test before marking this workstream complete.

**Supabase DB webhooks:** The `notify-partner-checkin` and `notify-todo-assigned` functions are triggered by DB webhooks (Supabase Dashboard → Database → Webhooks). The `notify-appointment-reminder` and `notify-kick-reminder` functions are triggered by pg_cron — enable the `pg_cron` extension in Supabase and create the cron jobs in the SQL Editor. Document both in `SUPABASE-SETUP.md`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mobile client → users table | Anon key + RLS; client can only UPDATE its own row |
| DB webhook → Edge Function | Supabase internal; payload is trusted but must be validated for shape |
| Edge Function → Expo Push API | Service-role context; payload must pass assertNoHealthData before dispatch |
| Edge Function → Supabase DB | Service-role bypasses RLS; queries must be scoped by household_id explicitly |
